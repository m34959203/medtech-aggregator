# Архитектура

Пять слоёв, два канала сбора, одна воронка нормализации.

```
┌─────────────────────────────────────────────────────────────────────┐
│ ИСТОЧНИКИ                                                             │
│   ① Push — клиника грузит прайс (xlsx/csv/pdf/скан)                   │
│   ② Pull — автосбор: веб-парсер сайтов + API-коннекторы (по cron)     │
└───────────────┬───────────────────────────────┬──────────────────────┘
                │ push                           │ pull
                ▼                                ▼
┌─────────────────────────────────────────────────────────────────────┐
│ СЛОЙ 1 · ПРИЁМ (Кейс 1)                                              │
│   file_parser   web_scraper   api_connector   scheduler              │
│                       │                                              │
│                       ▼                                              │
│   ★ НОРМАЛИЗАЦИЯ (общая воронка): fuzzy + LLM + дедупликация          │
└───────────────────────┬───────────────────────────────────────────--┘
                        ▼
┌─────────────────────────────────────────────────────────────────────┐
│ ЯДРО ДАННЫХ · PostgreSQL/SQLite                                      │
│   clinics · service_catalog · sources · ingestion_runs · prices      │
└───────────────────────┬───────────────────────────────────────────--┘
                        ▼
┌─────────────────────────────────────────────────────────────────────┐
│ СЛОЙ 2 · АГРЕГАТОР (Кейс 2) — FastAPI                                │
│   /api/search · /api/compare/{id} · фильтры город/цена/категория      │
└───────────────────────┬───────────────────────────────────────────--┘
                        ▼
┌─────────────────────────────────────────────────────────────────────┐
│ ВИТРИНА ПАЦИЕНТА · Next.js — поиск, сравнение, карта клиник           │
└─────────────────────────────────────────────────────────────────────┘
```

## Почему два канала
- **② Pull** быстро наполняет базу без участия клиник — пациенту сразу есть что сравнивать.
- **① Push** даёт точные, заверенные самой клиникой прайсы.
- Со временем клиники переходят с парсинга на официальную загрузку/API. `source_type` хранит происхождение каждой цены.

## ★ Нормализация (ядро ценности)
`backend/app/ingestion/normalizer.py`. Стратегия от дешёвого к умному:
1. **fuzzy-match** (rapidfuzz, `token_set_ratio`) против `canonical_name` и всех синонимов;
2. при уверенности ≥ порога — принимаем сразу, без сети;
3. в зоне неоднозначности и при наличии `GROQ_API_KEY` — **LLM** выбирает из top-5 кандидатов либо предлагает чистое эталонное имя + категорию;
4. иначе создаём новую запись справочника (catalog растёт сам), помечаем низкой уверенностью на ручную проверку.

Каждое успешно сопоставленное сырое имя добавляется в `synonyms` услуги — справочник самообучается.

## Дедупликация
`backend/app/ingestion/service.py`. Для пары (клиника, услуга) держим одну цену. Приоритет источника при конфликте: **upload > api > web_scrape** — официальная загрузка клиники не перезаписывается парсером. Внутри одной пачки дубли схлопываются по минимальной цене.

## Схема БД
- **clinics** — клиники (+ координаты для карты).
- **service_catalog** — единый справочник: `canonical_name`, `category`, `synonyms` (jsonb).
- **sources** — реестр источников (тип, url/endpoint, cron, enabled, last_run_at).
- **ingestion_runs** — журнал запусков приёма (channel push/pull, формат, статус, items_found).
- **prices** — цены: `service_id` (связь со справочником), `source_type`, `raw_name`, `price`, `match_confidence`, `valid_from`.

Две ключевые связки:
- `prices.service_id → service_catalog.id` — сравнение «одной и той же» услуги между клиниками.
- `prices.source_type` — происхождение цены (доверие + дедуп).

## Масштабирование
- Планировщик: `scheduler.py` (cron) → Celery + Redis beat в проде.
- Веб-парсер: httpx+BeautifulSoup для статики, Playwright-хук (`scrape_dynamic`) для SPA.
- БД: SQLite для мгновенного запуска → PostgreSQL для прода (один `DATABASE_URL`).
- Эмбеддинги: текущий fuzzy+LLM можно усилить векторным поиском (pgvector) для семантического матчинга.
